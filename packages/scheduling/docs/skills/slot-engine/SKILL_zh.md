# 时段引擎

`import { computeAvailableSlots } from "@agent-native/scheduling/core"`

## 输入
- 事件类型配置（时长、缓冲、最短提前通知、周期、限制、时段间隔）
- 日程（时区 + 每周可用性 + 日期覆盖）
- 忙碌时段（UTC，从外部日历 + 现有预约聚合）
- 按桶的预约计数（日/周/月/年）用于限制执行
- 要计算的时间范围
- 可选的每时段座位数 + 已占座位映射
- 查看者时区（用于限制分桶，如果与日程不同）

## 输出
- `Slot { start: ISO, end: ISO, available: boolean, seatsRemaining?, hostEmail? }` 数组

## 不变量

该函数按顺序保证：

1. 没有时段落在过去（`now + minimumBookingNotice` 是下限）。
2. 没有时段与忙碌时段重叠（应用了缓冲）。
3. 没有时段落在该天日程的可用时段之外。
4. 没有时段超过预约限制。
5. 没有时段落在事件的周期之外（滚动/范围）。
6. DST 安全：我们在做任何区间算术之前，使用日程的时区转换为 UTC。

## 调试

如果时段意外缺失：
1. 检查日程的每周可用性，查看日程时区（不是 UTC）中该星期几的情况（周末可能跨日期线偏移）。
2. 检查该本地日期的日期覆盖（空区间 = 完全阻止）。
3. 检查合并后的忙碌时段是否有重叠，*包括*前/后缓冲扩展。
4. 检查预约限制：单个现有预约可能关闭一整天。
5. 检查周期上限：滚动周期限制未来时段的显示范围。

如果时段意外存在：
1. 检查 `now` 是否正确传入（默认为 `new Date()`）。
2. 检查提供商的忙碌时段是否为 UTC。
3. 确认时段间隔符合预期（默认 = 时长）。
