# MVP 后续

## 规则

仅推荐或执行使当前 MVP 更真实、更已验证、更已文档化或更可发布的收尾工作。不要提议新的产品范围，除非存在 MVP 为真实用户工作的明确阻碍。

## 工作流

1. 识别当前功能或线程结果。
2. 按以下顺序检查未完成的工作：
   - 失败或跳过的验证
   - 未审查的真实数据试点结果
   - 待审查/提案/批准队列
   - 不再匹配行为的文档
   - 阻碍完整准备/发布卫生的无关脏文件
3. 推荐关闭这些缺口的最小下一批次。
4. 明确跳过诱人的膨胀：新集成、仪表板、设置、抽象或额外 UI，除非它们直接解除真实使用的阻碍。
5. 如果用户说"做吧"，在安全的地方并行运行收尾工作，并保持编辑最小化。

## 输出形状

以具体的下一步开头。保持列表简短。分开：

- **现在做**：验证、真实试点、文档或 Bug 修复。
- **推迟**：应等待真实用户反馈的有用想法。
- **阻碍**：任何需要的特定用户/手动操作。

## 验证偏好

当工作树有无关脏文件时，优先使用非变更检查。仅当完整 `pnpm prep` 不会重写其他人的并发工作时才使用它。
