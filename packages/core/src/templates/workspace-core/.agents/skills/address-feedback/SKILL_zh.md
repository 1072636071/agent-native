# 处理反馈

当用户分享反馈文档、问题、线程或粘贴的笔记并要求你处理反馈时，使用此技能。

默认姿态是判断加行动：修复你同意的清晰、已验证的 Bug；带理由提议 UX 变更；跳过或标记低信号、不清楚或超出范围的项目。

## 前提

- 如果没有提供链接或反馈文本，请求提供。
- 在修改代码之前阅读仓库的 `AGENTS.md`。
- 当可用时，使用来源的相应连接器/插件/技能，而不是抓取认证页面。

## 步骤

1. 阅读反馈来源。

   | 来源 | 读取方式 |
   | --- | --- |
   | Notion 链接 | Notion 连接器或 Notion 技能 |
   | Google Docs/Drive 链接 | Google Drive 连接器或 Google Docs 技能 |
   | Linear 链接 | Linear 连接器（如已安装）；否则请求粘贴内容 |
   | GitHub issue 或 PR | GitHub 连接器或 `gh issue view` / `gh pr view` |
   | Slack 线程 | Slack 连接器 |
   | 公共 URL | 网页浏览 |
   | 粘贴文本 | 直接读取 |

   仅对公共 URL 使用网页浏览。认证门控的文档通常需要其匹配的连接器。

2. 对每个可操作项目进行分类。

   - **Bug**：损坏的行为、崩溃、错误数据、死链接、包/API 不匹配或捕获的异常。当你同意时验证并修复。
   - **UX 建议**：设计、可发现性、工作流或功能反馈。除非用户明确要求你实现 UX 变更，否则先提议最干净的版本。
   - **问题或不清楚**：缺少细节、矛盾的反馈或你无法检查的行为。询问或标记。
   - **超出范围**：不在此仓库、已发布、有意不支持或信号太低。简要说明并跳过。

3. 当反馈看起来像错误时检查 Sentry。

   - 使用 Sentry 技能/插件（如果可用），或仓库中记录的 Sentry 脚本。
   - 按路由、堆栈符号、错误文本和症状关键词搜索。
   - 默认组织是 `builder-io`，除非用户指定其他。
   - 找到匹配时引用 issue ID 或链接。
   - 如果没有匹配，直说。

4. 仅修复你同意的明确 Bug。

   - 修复前验证：本地重现、阅读相关代码、检查日志或用堆栈跟踪确认。
   - 保持每个修复狭窄并映射到反馈项。
   - 遵循现有项目约定和附近模式。
   - 除非用户要求，否则不要切换分支、暂存、重置、强制推送或打开 PR。
   - 当 Bug 风险值得时添加或更新针对性测试。

5. 谨慎处理 UX 反馈。

   不要通过添加更多可见控件、帮助文本、横幅、顶级导航或默认始终打开的面板来解决 UX 问题。优先选择更干净的交互模型：

   - 使现有主要操作更可发现。
   - 移除竞争元素，使正确的操作突出。
   - 将次要操作移入 `DropdownMenu`、`Popover`、`Sheet`、`Collapsible` 或标签页。
   - 围绕一个清晰操作改进空状态。
   - 对可选或高级控件使用渐进式披露。

   提议 UX 变更时，写为：改什么、为什么有帮助、权衡。保持每个提议简短。

6. 验证变更后的行为。

   - 运行最小的相关测试或类型检查命令。
   - 对于 UI 修复，可行时在浏览器中验证并检查实际屏幕。
   - 如果无法运行有用的验证，说明原因。

## 报告格式

保持最终报告简短：

```md
## 已修复的 Bug
- [反馈项] - [变更内容，文件:行号]

## 已标记但未修复的 Bug
- [反馈项] - [原因]

## UX 建议
- [反馈项] -> [提议变更]

## 已跳过
- [反馈项] - [原因]
```

仅包含有内容的章节。用户可以阅读 diff；不要写第二个反馈文档。

## 避免

- 不要默认同意每个建议。
- 不要捆绑无关的清理。
- 不要在没有明确用户批准的情况下实现使重要屏幕更繁忙的 UX 变更。
- 当本地应用可以运行时，不要在没有浏览器验证的情况下声称 UI 变更已完成。
- 不要编造 Sentry 匹配、受影响用户或重现步骤。

## 相关技能

- `github:gh-address-comments` 用于 GitHub PR 审查线程。
- `github:gh-fix-ci` 用于失败的 GitHub 检查。
- `sentry:sentry` 用于生产错误调查。
- `frontend-design` 用于已批准的 UI 实现工作。
- `qa` 用于更广泛的浏览器验证。
