# 前端设计

此技能指导创建独特的、生产级的前端界面。实现真正可工作的代码，具有强大的产品判断力、出色的可访问性和清晰的视觉观点。

用户可能要求一个组件、页面、完整应用、仪表板、营销界面或重新样式化。在编码之前，了解受众并选择适合产品的方向，而非默认使用通用 SaaS 润色。

## 设计思考

在编码之前，决定：

- **目的**：这个界面使什么工作流更容易？主要操作是什么？
- **受众**：谁会反复使用它，应该感觉快速、平静、有趣、高端、编辑式、技术性还是实用？
- **调性**：选择具体的审美方向：精致极简、密集运营控制台、编辑式、趣味、工业、温暖手工、高对比度数据工具等。
- **信息层次**：前五秒内必须看到什么，什么应该逐步展示？
- **差异化**：什么使这个设计感觉是为这个确切领域量身定制的？

然后实现可工作的代码，使其具有凝聚力、可访问性、响应性和小细节上的精致：排版、间距、文案、动效、空状态、加载状态、焦点状态和错误状态。

## 极简主义和逐步展示

默认采用 Apple/Linear 级别的克制：使主要工作流显而易见，然后移除当前不帮助该工作流的所有内容。精致的 UI 通常比第一个合理实现有更少的可见控件、更少的边框、更少的标签和更少的解释性界面。

- **从减法开始**：在添加可见控件、横幅、工具栏行、卡片或解释性块之前，问什么可以被移除、合并、重命名或移入现有控件。
- **一个主要操作**：每个界面应该有一个主导的下一步操作。次要操作属于菜单、弹出框、命令面板、展示行或上下文悬停/焦点状态，除非它们被频繁使用。
- **逐步展示罕见工作**：高级选项、诊断、元数据、设置、导入/导出、破坏性操作和检查工具应保持隐藏直到被请求。优先使用带工具提示的小图标触发器、弹出框、抽屉或详情面板，而非永久性框架。
- **保持框架安静**：避免新的始终可见的栏、徽章、标注、帮助文本和计数器，除非它们防止错误或是重复使用的核心。状态通常可以是一个点、环、静音计数或工具提示。
- **内容优先于容器**：不要将每个部分包裹在卡片中。在添加框之前使用空白、对齐、排版、分隔线和全宽条带。
- **为反复使用设计**：生产应用 UI 在第一百次使用后应该感觉平静。如果控件大喊、动画、自我解释或为偶尔的操作占据整行，则隐藏或压缩它。
- **使缺失有意**：空状态应该稀疏且以行动为导向。不要因为屏幕感觉空白就用营销文案、装饰性艺术或功能列表填充空白空间。
- **使用熟悉的原语**：图标按钮需要清晰的工具提示。菜单、弹出框、标签页、开关和分段控件应该承载复杂性，而不是一次暴露每个选项。

## 审美指南

- **排版**：首先使用产品现有的排版系统。对于全新的公共页面，选择有特色但可读的字体，并保持尺寸适合界面。
- **颜色和主题**：使用语义令牌和 CSS 变量。避免单色调色板和默认的紫色/蓝色渐变，除非品牌要求。
- **动效**：优先使用有目的的过渡和小状态变更。使用 CSS transitions/keyframes，除非应用已使用动效库。
- **构图**：匹配工作流。运营应用应该密集且可扫描；营销或作品集页面可以更具沉浸感。
- **视觉资产**：网站、游戏和以对象为中心的页面在图像帮助用户理解主题时需要真实或生成的媒体。
- **响应式适配**：文本不得溢出按钮、卡片、标签页、侧边栏或固定格式工具。为面板、网格、工具栏和计数器使用稳定尺寸。

## Agent-Native UI 规则

- Agent-native 应用使用 React、Vite、Tailwind CSS、shadcn/ui 和 `@tabler/icons-react`。
- **使用 shadcn/ui 原语处理标准 UI**：`DropdownMenu`、`Popover`、`Dialog`、`AlertDialog`、`Sheet`、`Tabs`、`Tooltip`、`Select`、`Command`、`Sidebar`、`Table`、`Card`、`Badge`、`Skeleton` 及相关原语。
- **在接触 shadcn/ui 组件时，如果存在 `shadcn-ui` 技能也要阅读。** 该技能涵盖 `components.json`、CLI 文档、组件组合、主题和注册表工作流。
- 在导入 shadcn 组件之前检查 `app/components/ui/`。如果缺少原语，从应用根目录使用 `pnpm dlx shadcn@latest add <component>` 添加，然后审查生成的文件。
- 不要使用手动绝对定位和点击外部效果构建自定义下拉菜单、菜单、弹出框、模态框或确认框。
- 绝不使用浏览器对话框（`window.alert`、`window.confirm`、`window.prompt`）。使用 `AlertDialog`、`Dialog` 或应用特定的确认 UI。
- 所有第一方 UI 图标使用 Tabler 图标。不要添加 Lucide、Heroicons、内联 SVG 图标集或 emoji 图标。
- 使用 `@agent-native/core/client` 中的 `useActionQuery` 和 `useActionMutation` 处理 action 支持的 UI。标准 CRUD 应通过 action 进行，而非自定义 `/api/` 路由。
- 尽可能保持 UI 乐观：立即更新缓存和导航，然后在变更结果上协调或回滚。
- 自定义样式属于 Tailwind 类、组件 CSS 或现有全局 CSS 主题文件；避免内联样式。

## shadcn/ui 设计规则

- 首先使用内置组件变体（`variant`、`size`），然后覆盖类。
- 使用语义令牌（`bg-background`、`text-muted-foreground`、`border-border`、`bg-primary`）而非原始 Tailwind 颜色用于应用框架和可复用组件。
- 在 flex/grid 布局中使用 `gap-*` 而非 `space-x-*` 或 `space-y-*`。
- 当宽度和高度相等时使用 `size-*`，使用 `truncate` 而非拼写 overflow/ellipsis/nowrap。
- 使用本地 utils 别名中的 `cn()` 处理条件类。
- Dialog、Sheet、Drawer 和 AlertDialog 内容必须有可访问的标题。仅当可见设计已传达标题时才使用 `sr-only`。
- 将菜单/列表项放在其组原语内：`SelectGroup`、`DropdownMenuGroup`、`CommandGroup` 及等效项。
- 当内容有标题、描述、内容或操作时使用完整的 `Card` 组合。不要将复杂卡片倾倒到单个 `CardContent` 中。
- 小选项集使用 `ToggleGroup`，二进制设置使用 `Switch`，多选使用 `Checkbox`，多选一使用 `RadioGroup`，数值使用 `Slider`/输入。
- 对于表单，优先使用应用现有的 shadcn 表单模式。如果较新的 `Field`、`FieldGroup` 或 `InputGroup` 原语已安装或适合添加，使用它们而非原始布局 div。
- 加载状态使用 `Skeleton`、`Progress`、`Spinner` 或应用现有的加载原语。空状态应有一个清晰的下一步操作。

## 反模式

避免：

- 通用 AI 美学：紫色渐变、到处都是玻璃卡片、模糊的闪烁语言、装饰性斑点，以及脱离上下文的英雄区域。
- shadcn 原语的自定义重新实现。
- 当语义令牌或变体可行时，在共享组件上使用原始颜色覆盖。
- 为罕见操作添加新的始终可见控件。优先使用菜单、弹出框、抽屉、标签页、可折叠部分或高级区域。
- 全宽横幅、持久帮助行、装饰性卡片，或用于可以是紧凑控件的状态的解释性框架。
- 将逐步展示视为可选。如果控件不是主要日常工作流的一部分，在上下文、悬停、焦点或明确的用户意图使其相关之前隐藏它。
- UI 卡片嵌套在其他卡片内。
- 在悬停/加载时调整大小或移动固定格式 UI 的文本或图标。

## 验证

对于重要的前端工作：

1. 运行相关的格式化器/检查。
2. 当应用需要时启动开发服务器。
3. 使用桌面和移动宽度的浏览器截图验证。
4. 检查交互状态：悬停、焦点、加载、空、错误和破坏性确认。

## 相关技能

- **shadcn-ui** — shadcn CLI、组件文档、组合规则、主题和注册表
- **self-modifying-code** — 智能体可以编辑源代码以应用设计变更
- **storing-data** — 所有数据存储在 SQL 中；使用 action 进行数据访问
- **actions** — `useActionQuery`/`useActionMutation` hooks 用于前端数据获取
