# Plan 文档质量 — 单一真相来源

本文件是 canvas 下方 plan 文档的规范质量标准：它如何阅读、使用哪些块、如何呈现开放问题，以及交付前检查。在编写 plan 文档之前完整阅读；这是质量标准。不要从记忆中编写文档或按模式改写这些规则。

<!-- SHARED-CORE:document-quality START -->

**文档是严肃的技术 plan，不是营销。** 以一个强 Claude 或 Codex 实现 plan 的方式编写：结果优先、散文优先、自包含且具体。陈述目标和"完成"的含义、范围和非目标、提议的方法及其关键决策和理由、命名真实文件、符号、action 和数据形状的有序步骤、风险，以及一个结尾验证步骤（测试、构建或可检查的行为）。用具体内容替换模糊散文；永远不要交付像"让它工作"这样的步骤。没有英雄艺术、渐变、logo、导航栏、口号、价值主张、巨大的着陆页标题或营销卡片，除非用户明确要求。

**每个发布的 plan 必须独立。** 即使 agent 正在修订现有 plan，输出也是做工作的 plan，不是对话的变更日志。不要写像"保留之前的 plan"、"不要丢弃旧想法"、"如上所述"、"此修订"、"与先前版本不同"或"来自早期 plan 的更正"这样的短语。将正确的决策作为正常的目标、架构、范围和路线图散文折叠到 plan 中。一个从链接打开 plan 但没有聊天历史的审查者应该能理解它。避免仅在没有上下文时有意义的否定框架（"不是旧模式"、"不仅仅是 X"），除非对比在 plan 中定义且确实有帮助；直接陈述正面模型。

**使抽象 plan 立即可读。** 如果想法是广泛的、战略性的或面向第三方审查者的，在密集的架构、模式表、清单或路线图之前，在顶部附近放一个具体的产品快照。对于支持 UI 的概念，该快照通常是一个顶部 canvas 应用状态加一个简短段落，说明用户看到什么以及底层有什么变化。然后将机制、数据流、同步边界和实现细节放在单独的图或文档部分中。

**保留用户的抽象级别。** 一个激励用例不自动是架构。当提示描述更广泛的框架、产品模式或可复用原语时，将可复用核心与特定应用、provider、客户、脚本或启动示例分开。使用具体示例使 plan 可理解，然后明确哪些部分是核心、哪些是应用特定适配器、哪些是未来示例。

**当顶部视觉存在时，它们和文档永远不重复彼此。** 对于 UI 工作，UI 故事存在于顶部视觉表面：用于静态检查的 canvas 画板，加上当流程应该是功能性时的 prototype 标签页。文档承载视觉无法展示的技术深度 — 具体的文件/符号映射、API 和数据契约、代码片段、迁移或实现阶段、风险和验证。对于架构/代码审查，反转：文档是视觉表面，每个推荐携带自己附近的内联 `diagram` / `data-model` 块加文件证据（下方 `diagram` 要点拥有如何编写这些图）。仅在真正新的细节视图或比较时在文档中重复线框图。对于非视觉工作完全跳过视觉表面，写一个干净的富文档。对于简单的二元 UI 视觉选择，仅在 canvas 中展示两个方向；不要将相同选项作为正文线框图或散文重复。将实际选择放在底部"Open Questions"表单中。

**使用正确的块，并让它承载实质。** 对于权威的、机器检查的块类型及其数据 schema 列表，调用 `get-plan-blocks` — 它返回实时注册表词汇（类型、MDX 标签、位置、关键字段），因此你永远不会发出编辑器无法渲染或往返的块：

- `rich-text` 用于带真实粗体/斜体/代码/链接和嵌套列表的 plan 散文。
- `annotated-code` 用于文件映射：当一个承重文件值得突出时，优先使用带注释的演练而非裸 `code` 块 — 承载真实的、语法高亮的代码并将简短的边注锚定到实际更改的行（新 action、更改的 schema、连接点），因此读者看到什么重要以及为什么，而不是为代码而代码。每个注释是 `{ lines: "12" | "12-18"; label?; note }`；每个文件保持几个高信号注释，而不是每行一个。仅突出值得阅读的文件；永远不是每个触及文件的详尽列表，也永远不是仅散文的文件描述。仅当没有需要指出的内容时才降级到普通 `code` 块用于一次性片段。当多个文件重要时，将块分组在垂直 `tabs` 块（标准标签原语）中而非自定义容器。如果确切代码未知，展示最小的合理计划形状或注释存根命名要填写的内容。（`code-tabs` 和 `implementation-map` 是遗留的：它们的渲染器为旧 plan 保留，但不要编写新的。）
- 对于决策：如果审查者仍必须在真正开放的二选一之间选择，将其作为 `single` 问题放在底部 Open Questions `question-form` 中 — 每个真实替代一个选项，每个带简短细节和你选择的 `recommended: true`；不要在其他地方重述相同选择。如果你已经承诺了一个方法，将其陈述为已定的散文或带 `tone="decision"` 的 `callout`，可选地加一个 `columns` 块用于你权衡的选项的并排比较 — 而不是你已经回答的问题的令人困惑的文档中间表单。
- `columns` 用于并排前后或当前/目标比较，每侧需要真实嵌套块；清晰标记列，当平行阅读是重点时避免垂直堆叠比较块。
- `diagram` 用于二维架构、依赖、数据流或状态关系，仅当它澄清真实内容时。优先使用标准二维布局 — 配对的前后面板、分层图、泳道、依赖映射、矩阵或分组区域；不要默认从左到右链，仅在关系真正是序列时使用线。对于架构/代码图，优先使用带语义 HTML 和内联 SVG 的 `data.html` / `data.css`，以便图可以直接使用面板、层、矩阵、箭头、注释和响应式布局。使用渲染器拥有的原语如 `.diagram-panel`、`.diagram-card`、`.diagram-node`、`.diagram-box`、`.diagram-pill`、`.diagram-muted` 和 `[data-rough]` 编写图 HTML；它们通过 `--wf-ink`、`--wf-muted`、`--wf-line`、`--wf-paper`、`--wf-card`、`--wf-accent`、`--wf-accent-soft`、`--wf-warn` 和 `--wf-ok` 映射到 plan 的 Tailwind 主题变量，并在 sketchy 模式下切换到 Excalifont 加 rough.js 轮廓。不要设置 `font-family`，不要在图 HTML 或 CSS 中硬编码十六进制、rgb 或 hsl 颜色。为 sketch 字体留空间：保持标签简短，给节点充裕的宽度，将边界/注释标签放在未使用的空间而非节点上方；标签不得与节点、连接器或彼此重叠。对于现有 HTML 图的小文本/SVG 更改，使用带唯一 `find`/`replace` 片段的 `patch-diagram-html` 而非重新发送整个 `data.html` 字符串。仅对小型预览或真正顺序流程使用遗留 `nodes` / `edges`。在架构/代码 plan 中，优先使用重复的节拍节奏：推荐标题、信心和类别徽章、代码路径证据、本地前后或当前/目标空间图，然后简洁的 Problem/Solution/Why 文本。
- `tabs` 用于多个状态、方向或比较。仅揭示散文的标签通常意味着 plan 规格不足 — 除非标签有意仅用于文档，否则包含相关视觉。
- `table`、`checklist`、`callout` 用于可扫读的结构。

**开放问题在答案会改变 plan 时作为表单存在于底部。** 在最终的 `question-form` 块中呈现可回答的未解决决策，标题为"Open Questions"，以便渲染器将其呈现为独立部分。该底部表单是唯一列举开放问题的地方：永远不要在文档前面添加第二个"Open Questions"标题、列表或相同问题的重述。概述散文中的一行指针（"几个决策仍然开放 — 见下方 Open Questions"）是可以的，但不要在上面复制问题列表或平行的 questions/decisions 部分。对明确选择使用 `single` 或 `multi`，对约束使用 `freeform`，对你选择的默认使用 `recommended: true`，仅在选项尚未在顶部 canvas 中可见时使用选项 `wireframe` / `diagram` 预览。`single` 和 `multi` 问题总是渲染一个写入字段，以便审查者可以用自定义选项回答 — 永远不要自己添加显式的"Other"选项；仅当自由文本答案无意义时设置 `allowOther: false`。将不可回答的假设或风险保持为相关部分中的简洁 `callout` 块。永远不要在 plan 叙述中埋藏 questions/decisions 墙，永远不要两次问相同问题。

对于复杂 plan，不要在没有开放问题审计的情况下结束。如果架构、范围、UX、数据形状、推出、provider 映射或所有权仍取决于选择，要么用理由承诺推荐，要么将其添加到底部表单并带推荐默认。一个没有开放问题的复杂 plan 仅当每个有意义的决策都已明确做出时才可以。

**验证必须演练真实工作流。** 当 plan 更改 UI、本地文件、同步、provider、浏览器行为或多应用流程时，最终验证部分应超越类型检查/单元测试。至少包含一个匹配用户旅程的端到端冒烟测试，如新的仓库/文件夹、真实的清单或数据固件、浏览器交互、保存/同步操作，以及磁盘或数据库断言。当已知时命名命令或手动浏览器路径。

**`custom-html` 仅是有界逃生舱** — 块内的单个完整片段，永远没有 `html`/`head`/`body`/`script` 标签，永远没有通用占位符、密度演示或自定义 HTML 工作的证明。正常 plan 优先使用原生块。对于架构/代码审查，使用 `diagram` `data.html` / `data.css` 处理丰富的本地 HTML/SVG 图而非 `custom-html`。对于 UI/产品工作，`custom-html` 永远不是请求的 mockup、UI 状态或视觉比较的主要归宿。如果 UI 保真度需要 HTML/CSS、图像捕获或真实 React/CSS，产品修复是 canvas 对该产物类型的支持，而不是将 mockup 移入文档。

**在交付前，打开 plan 并检查它。** 在请求批准前修复重叠、过多空白、裁剪片段、误导性非活动控件、对比度差和不可读图。
