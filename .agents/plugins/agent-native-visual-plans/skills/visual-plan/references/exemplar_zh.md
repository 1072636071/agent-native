# 好与坏的示例 — 单一真相来源

本文件是一个优秀 plan 的规范工作示例（以及要避免的反模式）。在编写 plan 之前，与 document-quality 和 canvas 参考一起阅读；这是这些 plan 必须达到的标准。

**好的。** 一个 todo 应用的 UI 优先 plan：一个带 `desktop` 画板的 canvas，其 `data.html` 是真实的 flex 布局 — 一个链接侧边栏（`Inbox 12`、`Today 4`、`Done`），一个带 `<h1>Today</h1>` 的主列，accent `.wf-pill` 用于过滤器，一个柔和的节标签 `OVERDUE`，以及 `.wf-card` 任务行承载真实标题、截止日期和一个 primary `button.primary` — 仅通过裸元素、辅助类和 `--wf-*` token 样式化，因此渲染器应用正确的桌面足迹、主题和微妙的整体帧摆动。纯文本设计者笔记间隔放置在帧外，仅指向需要解释的控件。在它下方，一个 Claude/Codex 级别的文档：目标和完成标准，几个 `code` 块（当超过一个时分组在垂直 `tabs` 块中）展示承重文件的真实形状，一个带 `tone="decision"` 的 `callout` 陈述所选方法，后面跟一个 `columns` 块权衡两个真实选项，以及一个验证步骤 — 这些都不重复 canvas。如果任务还改变了多步骤完成流程，相同的顶部区域包含一个 Prototype 标签页，其屏幕使用与 canvas 画板相同的标签和状态，带有 `data-goto` 控件用于序列。这就是标准。

**好的。** 一个广泛的产品架构 plan 以一个简单的建议和一个具体的应用状态在抽象之前开始。第一个 canvas 画板是匹配当前应用 shell 的纯产品 UI；附近的笔记解释用户可见的增量。下面一个单独的图显示机制，如文件或数据流。然后文档将可复用核心与应用/provider 适配器和示例分开，涵盖契约、文件夹或 schema 形状、同步边界、路线图、非目标、一个底部 Open Questions 表单用于未解决的决策，以及一个验证部分，至少包含一个现实的端到端冒烟测试。一个未参与聊天的审查者从顶部快照就能理解想法，然后再阅读技术 plan。

**好的。** 一个后端架构审查的 `/visual-plan`：没有顶部 canvas。文档以上下文和图例开始，然后重复推荐卡片：标题、信心/类别徽章、真实文件路径的等宽网格、一个内联二维前后或分层架构图，以及使用代码库词汇的简洁 Problem/Solution/Why 要点。图使用空间显示边界、层和所有权；它不是默认的从左到右链。plan 以顶部推荐结束，仅当下一个架构方向真正开放时才有底部 question-form。这比顶部 canvas 更好，因为每个图都局部于它支持的主张。

**坏的。** 一个 `data.html` 带硬编码的十六进制颜色、`font-family` 或固定像素宽/高；灰色占位符条在非骨架帧上"暗示"文本；为弹出框强制桌面 + 移动对；浮动带边框的注释卡片紧贴帧；一个手工编写的 kit-tree `screen` 而非 `html`；一个多步骤 UI 流程只有静态帧没有 prototype 标签页；一个 mockup 逃逸到文档 `custom-html` 块中；以及一个营销风格的文档带有英雄标题和价值主张，只是重述 canvas 已经展示的内容。同样坏的：一个仅架构的 plan 被强制放入带重叠文本的标签框顶部 canvas，而实际代码证据和推荐存在于其他地方；一个产品线框图将真实屏幕与仓库名、文件契约箭头、架构解释或虚构的永久检查器混合在一起；以及一个 plan 将自己描述为先前对话的修订而非独立提案。永远不要产生这些。
