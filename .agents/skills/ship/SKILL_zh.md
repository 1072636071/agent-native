# Ship

端到端交付当前分支：提交并推送所有本地工作，打开就绪 PR，运行 `/babysit-pr`，当看护合并门控满足时合并，然后在合并落地后运行 `/new-branch`。

## 不可协商的交付不变量

`/ship` 交付的是**分支**，不仅仅是 agent 自己的编辑。提交并推送当前分支上所有非 gitignore 的本地更改，包括用户或其他并发 agent 创建的工作。不要因为你没有编写它们就留下本地更改。唯一的常规例外是 `learnings.md` 和被忽略/个人文件。

调用 `/ship` 是合并此 PR 的明确授权，当下方合并门控通过时，除非用户说不合并。不要为了合并一个干净的 PR 再次询问。不要在创建 PR 后停止；`/ship` 的默认结果是一个已合并的 PR 和一个合并后的新分支。

如果分支更新了模板或可发布的包，交付不会在合并时停止。仅当受影响的模板在生产环境中上线且受影响的包已成功发布/发布后，才将工作视为已交付。如果生产模板部署或包发布失败，当现有代码已包含修复时重新触发失败的作业；否则进行必要的代码/配置修复并交付该后续，直到生产环境上线。

## 步骤

1. **留在当前分支**：在打开 PR 之前切勿创建、切换、变基、重置或暂存。此仓库使用共享/平台管理的分支；交付你已经所在的分支。

2. **检查本地更改**：运行 `git status --short` 和 `git diff --stat` 以了解所有修改/未跟踪的文件。多个 agent 可能已添加工作；在交付中包含所有非 gitignore 的本地文件，而不是暂存、跳过或回退它们。

3. **验证足以避免明显损坏**：对更改区域运行针对性测试。当实际时运行 `pnpm run prep`。如果 prep 缓慢、不稳定或被并行的进行中编辑污染，不要阻塞交付：推送并让 GitHub Actions 成为 `/babysit-pr` 监控的验证门控。

4. **暂存并提交**：暂存所有更改/未跟踪的文件，除了 `learnings.md` 或 gitignore 的个人文件。基于实际差异编写简洁、描述性的提交消息。切勿添加 `Co-Authored-By` 或其他 agent 归属。

5. **推送**：推送当前分支。如果分支没有上游，用 `git push -u origin <branch>` 设置。

6. **打开或更新就绪 PR**：使用当前分支。PR 默认就绪供审查，不是草稿。不要在标题/正文中放置 `codex`、`[codex]` 或类似的 agent 标签。

7. **立即看护**：运行 `/babysit-pr <number>` 并严格遵循该技能的循环。将 `babysit-pr` 视为如何观察 PR 的真相来源；不要复制、缩短或发明更轻的监控循环。其步骤 0 是权威的：每个循环从提交并推送所有本地文件和任何未推送的提交开始，然后检查可合并性、每个未处理的审查评论的回复状态和 CI。持续进行直到 PR 合并/关闭或用户明确告诉你停止。

8. **在允许时合并**：因为 `/ship` 包含合并授权，仅在 `/babysit-pr` 的合并要求连续 10 分钟同时为真时用 `gh pr merge <number> --squash --admin` 合并：干净的工作树、没有未推送的提交、GitHub Actions 绿色、所有审查评论已处理/回复、且可合并。

9. **需要时验证生产环境已上线**：如果分支更改了 `templates/*`、发布模板的文档/站点、或任何影响模板的部署配置，验证受影响的模板生产部署成功完成且线上站点正在提供新构建。如果部署因临时基础设施/构建拾取问题而失败，重新触发；如果因代码、配置、依赖或生成文件问题而失败，修复问题并交付后续。如果分支更改了可发布的包如 `packages/core`、`packages/dispatch`、`packages/scheduling`、`packages/pinpoint` 或 `packages/skills`，验证发布/发布工作流完成且包版本可从注册表或包主机获取。重新触发临时发布失败；修复并交付代码/配置失败。

10. **合并后创建下一个分支**：PR 合并且 `origin/main` 包含合并提交后，运行 `/new-branch`。遵循该技能的预检、暂存门控、分支命名和暂存报告规则。这是交付流程中唯一的分支移动。

11. **报告**：总结 PR URL、合并结果、新分支名称、验证、适用时的生产部署/发布验证、以及处理的任何反馈/CI 修复。

## 重要

- **多个 agent 并发运行。** 通常会有你未生成的本地更改文件。这是正常的。包含所有内容并继续前进。不要回退其他 agent 的工作；如果 CI 或审查反馈标记了真正的 bug 则修复它们。
- 切勿提交 `learnings.md` 或 `.gitignore` 中的文件。
- 如果反馈出现在内联评论或审查正文中，每个项目在合并前都需要修复或回复。
- 将 `/babysit-pr` 视为 CI/审查监控节奏、评论处理、本地文件推送纪律和合并门控的真相来源。如果观察者行为更改，先更新 `babysit-pr`。
- 当模板或可发布包更改时，将生产部署/发布验证视为 `/ship` 的一部分。绿色 PR 不够，如果受影响的模板构建或包发布后来失败。
- 将 `/new-branch` 视为成功合并后强制执行的，以便工作区在新的 `main` 上为下一个任务做好准备。
