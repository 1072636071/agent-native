# 处理反馈

当用户分享反馈文档、issue、线程或粘贴的笔记并要求你处理反馈时，使用此 skill。

默认姿态是判断加行动：修复你同意的明确、已验证的 bug；提出有理由的 UX 变更；跳过或标记低信号、不清楚或超出范围的项目。

## 前提条件

- 如果未提供链接或反馈文本，请索要。
- 在修改代码之前阅读仓库 `AGENTS.md`。
- 当可用时，使用相关连接器/插件/skill 作为来源，而不是抓取认证页面。

## 步骤

1. 阅读反馈来源。

   | 来源 | 读取方式 |
   | --- | --- |
   | Notion 链接 | Notion 连接器或 Notion skill |
   | Google Docs/Drive 链接 | Google Drive 连接器或 Google Docs skill |
   | Linear 链接 | Linear 连接器（如已安装）；否则要求粘贴内容 |
   | GitHub issue 或 PR | GitHub 连接器或 `gh issue view` / `gh pr view` |
   | Slack 线程 | Slack 连接器 |
   | 公共 URL | 网页浏览 |
   | 粘贴文本 | 直接阅读 |

   仅对公共 URL 使用网页浏览。需要认证的文档通常需要其匹配的连接器。

2. 对每个可操作项目进行分类。

   - **Bug**：损坏的行为、崩溃、错误数据、死链接、包/API 不匹配或捕获的异常。验证并在你同意时修复。
   - **UX 建议**：设计、可发现性、工作流或功能反馈。除非用户明确要求你实施 UX 变更，否则先提出最简洁的版本。
   - **问题或不清楚**：缺少细节、矛盾反馈或你无法检查的行为。询问或标记。
   - **超出范围**：不在此仓库中、已发布、故意不支持或信号太低。简要记录并跳过。

3. 当反馈看起来像错误时检查 Sentry。

   - 使用 Sentry skill/插件（如可用），或仓库的 Sentry 脚本（如有文档）。
   - 按路由、堆栈符号、错误文本和症状关键词搜索。
   - 默认组织是 `builder-io`，除非用户指定其他。
   - 找到匹配时引用 issue ID 或链接。
   - 如果没有匹配，直接说明。

4. 仅修复你同意的明确 bug。

   - 修复前验证：本地复现、阅读相关代码、检查日志或用堆栈跟踪确认。
   - 保持每个修复窄化并映射到反馈项目。
   - 遵循现有项目约定和附近模式。
   - 除非用户要求，否则不要切换分支、暂存、重置、强制推送或打开 PR。
   - 当 bug 风险值得时添加或更新针对性测试。

5. 审慎对待 UX 反馈。

   不要通过添加更多可见控件、帮助文本、横幅、顶级导航或始终打开的面板来解决 UX 问题。优先使用更简洁的交互模型：

   - 使现有主要操作更可发现。
   - 移除竞争元素，使正确的操作突出。
   - 将次要操作移入 `DropdownMenu`、`Popover`、`Sheet`、`Collapsible` 或标签页。
   - 围绕一个清晰操作改进空状态。
   - 对可选或高级控件使用渐进式展示。

   提出 UX 变更时，写为：改什么、为什么有帮助、权衡是什么。保持每个提案简短。

6. 验证变更行为。

   - 运行最小的相关测试或类型检查命令。
   - 对于 UI 修复，可行时在浏览器中验证并检查实际屏幕。
   - 如果无法运行有用的验证，说明原因。

## 报告格式

保持最终报告简短：

```md
## 已修复的 Bug
- [反馈项目] - [变更内容，file:line]

## 已标记但未修复的 Bug
- [反馈项目] - [原因]

## UX 建议
- [反馈项目] -> [提议变更]

## 已跳过
- [反馈项目] - [原因]
```

仅包含有内容的章节。用户可以阅读 diff；不要写第二份反馈文档。

## 避免

- 不要默认同意每个建议。
- 不要捆绑无关的清理。
- 不要在没有用户明确批准的情况下实施使重要屏幕更繁忙的 UX 变更。
- 当本地应用可以运行时，不要在没有浏览器验证的情况下声称 UI 变更已完成。
- 不要编造 Sentry 匹配、受影响用户或复现步骤。

- `github:gh-fix-ci` 用于失败的 GitHub 检查。

- `frontend-design` 用于已批准的 UI 实现工作。
- `qa` 用于更广泛的浏览器验证。
